## Recommended Cleanup Shape

The next agent should probably run a blueprint-style planning cleanup, but not
blindly regenerate everything. The existing plan files already contain detailed
implementation instructions. The cleanup should mostly reconcile, normalize,
and publish.

Suggested destination model:

1. Create one parent GitHub issue that acts as the planning index / PRD.
2. Create one GitHub issue per executable plan, probably issues for plans
   001-008.
3. Apply `ready-for-agent` to issues that are sufficiently specified and not
   blocked by another issue.
4. Preserve dependencies in issue bodies and/or comments.
5. Decide whether `docs/plans/` remains as a local mirror, is reduced to an
   index pointing to GitHub, or is removed after issue publication.

## Blueprint Guidance For The Next Agent

Use the `blueprint` skill as guidance for structure, but this handoff was not
produced by running the full workflow.

Recommended blueprint-like process:

1. Reconfirm the repo state:
   - `git status --short`
   - `git diff --name-status -- plans docs/plans`
   - `gh issue list --state all --limit 100`
2. Read:
   - `docs/agents/issue-tracker.md`
   - `docs/agents/triage-labels.md`
   - `docs/plans/README.md`
   - each numbered file in `docs/plans/`
3. Reconcile stale paths:
   - Replace `plans/README.md` references with `docs/plans/README.md`, unless
     the repo chooses to move plans back to `plans/`.
4. Convert local plans into hosted issues without losing:
   - priority, effort, risk, dependency, planned commit, STOP conditions,
     verification commands, and done criteria.
5. Create or update a local index that points to the GitHub issues.
6. Keep source code untouched. This cleanup is planning/tracker work only.
